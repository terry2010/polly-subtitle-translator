## 一、macOS 27 生成字幕功能概述

### 功能拆解

| 子功能 | 原理 | 平台支持 |
| --- | --- | --- |
| Speech Transcription（语音转写） | 英语音频 → on-device Speech-to-Text → 英文字幕 | iOS 27 / macOS 27 / tvOS 27 / visionOS 27 |
| Language Translation（语言翻译） | 英语字幕 → on-device Translation 模型 → 其他语言字幕 | iOS 27 / macOS 27（tvOS/visionOS 暂不支持） |

### 关键特性

- **完全本地处理**：音频不上传，隐私安全
- **零代码集成**：使用 `AVPlayerView`（macOS）/ `AVPlayerViewController`（iOS）的 app 自动支持
- **已有字幕优先**：视频本身有字幕轨时用原有的，不生成
- **支持内容**：HLS 直播、VOD 点播、本地文件、app 内视频
- **源语言**：目前仅英语
- **目标语言**：约 12 种（中/日/韩/法/德/意/西/丹/荷/挪/葡等）
- **系统设置路径**：设置 → 辅助功能 → 字幕和隐藏式字幕

### 平台差异

| 平台 | 语音转写 | 语言翻译 | 区域限制 |
| --- | --- | --- | --- |
| macOS 27 | ✅ | ✅ | 无 |
| iOS 27 | ✅ | ✅ | 无 |
| tvOS 27 | ✅ | ❌ | 仅美国/加拿大英语 |
| visionOS 27 | ✅ | ❌ | — |

### 播放器兼容性

| 播放器 | 内核 | 支持生成字幕 |
| --- | --- | --- |
| QuickTime / TV app / Safari | AVPlayer | ✅ 自动支持 |
| Infuse（Mac & tvOS） | 自定义 | ❌ 大概率不支持 |
| IINA | mpv | ❌ 不支持（mpv 不走 AVPlayer 管线） |
| VLC | 自研 | ❌ 不支持 |
| Movist Pro | 自定义 | ❌ 大概率不支持 |

**关键矛盾**：能自动生成+翻译字幕的播放器（QuickTime/TV app）格式支持差，打不开 mkv；能打开 mkv 的播放器（Infuse/IINA/VLC）用自定义内核，不支持系统生成字幕。

## 二、对项目各场景的影响评估

### 被 macOS 27 覆盖的场景（项目会失去这部分用户）

| 场景 | macOS 27 怎么做 | 项目是否还需要 |
| --- | --- | --- |
| 看 mp4 英语电影，想要中文字幕 | QuickTime 实时生成+翻译 | ❌ 不再需要 |
| 看 Safari 网页英语视频，想要中文字幕 | Safari 自动生成 | ❌ 不再需要 |
| 看英语 YouTube，想要中文字幕 | Safari 里自动生成 | ❌ 不再需要 |

### macOS 27 覆盖不了的场景（项目仍有价值）

| 场景 | macOS 27 的局限 | 项目优势 |
| --- | --- | --- |
| 用 IINA/Infuse/VLC 看 mkv | 自定义播放器不支持生成字幕 | 处理后的 mkv 任何播放器都能用 |
| 非英语视频（日韩法德） | 只能从英语生成/翻译 | 支持任意源语言 |
| 字幕文件处理/编辑/导出 | 不输出 srt/ass 文件 | 提取/导出/编辑 srt/ass |
| 批量处理整个剧集库 | 只能播放时实时生成 | 批量处理 + watch 模式 |
| 嵌回 mkv 给 Apple TV | 无法修改视频文件 | 合并回 mkv，跨设备使用 |
| 翻译质量可控 | 只能用 Apple 翻译模型 | 百度/Google/Bing/DeepL 可选 |
| 双语字幕 | 不确定是否支持 | 原文/译文/双语三种模式 |
| Apple TV 上的中文字幕 | tvOS 不支持翻译 | 提前在 Mac 处理好，Apple TV 播放 |
| 离线提前处理 | 实时生成有延迟和 CPU 开销 | 提前处理好，播放时零开销 |

### tvOS 趋势预测

| 时间 | tvOS 能力 | 项目 Apple TV 链路价值 |
| --- | --- | --- |
| 2026（tvOS 27） | 语音转写（仅英语，仅美/加），无翻译 | ✅ 唯一解 |
| 2027-2028（tvOS 28 预测） | 大概率加上语言翻译 | ⚠️ 但 Infuse 用自定义播放器，大概率仍不支持系统生成字幕 |
| 长期 | 多源语言支持 | ⚠️ 实时翻译可用，但批量/编辑/嵌回仍需要项目 |

**核心判断**：只要 Apple TV 用户还在用 Infuse/VLC 看下载的 mkv，项目就有价值。Apple 的系统级功能打不到自定义播放器的用户群。

## 三、三个候选方案

### 方案 A：放弃语音转写，强化文件处理

**核心思路**：不与 Apple 系统功能正面竞争，把所有资源投入到 Apple 不做的"字幕文件全生命周期管理"。

**功能清单**：

| 功能 | 说明 | Apple 是否做 |
| --- | --- | --- |
| 批量处理 | 一次导入整个剧集库，队列处理 | ❌ |
| Watch 模式 | 监控文件夹，新视频自动处理 | ❌ |
| 字幕质量工具 | 校对、术语表、上下文翻译 | ❌ |
| NAS/Docker 服务版 | 无头运行，7×24 自动处理 | ❌ |
| Jellyfin/Emby 插件 | 服务端自动翻译，对 Infuse 透明 | ❌ |
| 多源语言增强 | 日韩法德，macOS 27 的盲区 | ❌ |

**优势**：
- 每个功能都是 Apple 永远不会做的领域
- 直接增强现有"提取→翻译→编辑→合并"链路
- 投入产出比最高
- 和 macOS 27 零冲突

**劣势**：
- 放弃了语音转写能力，功能维度比 macOS 27 少一项
- 对"无字幕视频"场景无覆盖（只能处理已有字幕的视频）

**适合场景**：项目定位为"字幕文件处理工具"，目标用户是 NAS/剧集库管理者/Apple TV 用户。

---

### 方案 B：做语音转写但定位不同

**核心思路**：做语音转写，但不做"实时播放转写"（那是 macOS 27 的领域），只做"离线批量转写输出 srt 文件"。

**功能清单**：

| 功能 | 说明 | 与 macOS 27 的差异 |
| --- | --- | --- |
| 离线批量转写 | 批量转写视频音频，输出 srt 文件 | macOS 27 不输出文件 |
| 多语言转写 | 支持日语/韩语/法语等非英语源 | macOS 27 仅英语源 |
| 转写结果编辑 | 转写后的字幕可编辑校对 | macOS 27 不可编辑 |
| 转写+翻译一条龙 | 转写后直接翻译，输出双语 srt | macOS 27 不输出文件 |
| 嵌回 mkv | 转写+翻译的字幕嵌回视频 | macOS 27 无法修改文件 |

**优势**：
- 填补 macOS 27 的空白（不输出文件、仅英语源）
- 对"无字幕视频"场景有覆盖
- 差异化明确："macOS 27 生成字幕的文件版补充"

**劣势**：
- 受众比方案 A 小（需要"转写成文件"的用户少于"处理已有字幕"的用户）
- 技术成本高（集成 whisper.cpp、模型管理、GPU 加速适配）
- 模型体积大（几百 MB ~ 几 GB），增加 app 体积
- macOS 27 后续可能扩展更多源语言，缩小方案 B 的优势窗口

**适合场景**：项目想覆盖"无字幕视频"用户群，且愿意承担较高的技术成本。

---

### 方案 C：混合方案

**核心思路**：以方案 A 为主力，语音转写降级为辅助功能，只做"离线批量输出 srt"，不做实时播放转写。

**功能清单**：

```
二期核心（方案 A 全部）：
  ├─ 批量处理（一次导入整个剧集库）          ← 主力
  ├─ Watch 模式（监控文件夹自动处理）         ← 主力
  ├─ 字幕质量工具（校对、术语表、上下文翻译）  ← 主力
  ├─ NAS/Docker 服务版（无头运行）            ← 主力
  ├─ Jellyfin/Emby 插件（服务端自动翻译）     ← 主力
  └─ 多源语言增强（日韩法德）                 ← 主力

二期辅助（方案 B 子集）：
  └─ 语音转写只做"离线批量输出 srt"           ← 辅助
      → 不做实时播放转写（不和 macOS 27 竞争）
      → 只做"转写成文件"这一件事
      → 定位是"macOS 27 生成字幕的文件版补充"
      → 优先级低于核心功能，资源允许时再做
```

**优势**：
- 主力方向和 Apple 零冲突
- 辅助方向有差异化，不正面竞争
- 灵活：语音转写可以推迟，等 macOS 27 普及后看用户反馈再决定
- 既有文件处理护城河，又覆盖了"无字幕视频"场景

**劣势**：
- 范围最大，工作量最重
- 需要明确优先级，避免资源分散
- 语音转写部分的技术成本依然存在

**适合场景**：资源充足，想同时覆盖"已有字幕"和"无字幕"两类用户，但不想和 Apple 正面竞争。

## 五、推荐结论

**推荐方案 C（混合方案），以方案 A 为核心，语音转写降级为辅助。**

### 推荐理由

1. **方案 A 的所有功能都是 Apple 不做的领域**，是确定的护城河，应作为二期核心
2. **语音转写完全放弃太可惜**——"无字幕视频"是真实需求，且 macOS 27 有明确盲区（不输出文件、仅英语源）
3. **语音转写不做实时播放转写**——那个场景 macOS 27 免费送了，追不上也没必要追
4. **语音转写只做"离线批量输出 srt"**——这是 macOS 27 做不到的，有明确差异化
5. **语音转写优先级低于核心功能**——等核心功能做完、macOS 27 普及后看用户反馈再启动，降低风险

### 执行原则

```
1. 先做方案 A 的全部功能（批量/watch/NAS/插件/多源语言）
2. 语音转写作为 P3 优先级，资源允许时启动
3. 语音转写只做离线批量，不做实时播放
4. 语音转写优先支持非英语源（日语/韩语），这是 macOS 27 最大的盲区
5. 持续观察 macOS 27 后续版本扩展情况，动态调整语音转写的优先级
```
